# iOS Phase 0 — Compile + Test on iOS (feasibility proof)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Prove the APTrade codebase compiles for iOS and its 190 tests pass on the iOS simulator — *without* breaking the working macOS app. This is a feasibility proof, not a shippable iPhone UI (layout is Phase 2+).

**Approach:** Add `.iOS` to the package, gate the 6 AppKit touchpoints behind `#if` (cross-platform replacement where trivial, macOS-only `#if os(macOS)` + an iOS compile-path/stub otherwise), then build + test via `xcodebuild` against an iOS simulator. The three lower layers are already iOS-clean except one file.

**Reference:** `docs/ios-port-readiness.md` (the audit).

## Global Constraints

- **macOS must keep working at every step.** After each task, `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer swift build` and `… swift test` (macOS) must stay green (190/190). The macOS app is the source of truth; iOS support is additive.
- **Toolchain:** Xcode 26.5 at `/Applications/Xcode.app`; iPhoneSimulator SDK present; an iOS runtime is being installed (`xcodebuild -downloadPlatform iOS`). All `xcodebuild`/`swift` commands run with `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer`.
- **Phase 0 = compile + test, not feature parity.** Where an AppKit feature (PDF export styling, brand-image recoloring, save panel) has no trivial iOS equivalent, an iOS compile-path may be a minimal stub (e.g. return the base image / an unstyled PDF / a no-op) — clearly marked `// iOS Phase 0 stub — full impl in Phase 1`. Do NOT delete or weaken the macOS path.
- **Gating idiom:** prefer a genuine cross-platform API when one exists (e.g. SwiftUI `openURL` replaces `NSWorkspace.shared.open` on both platforms — no `#if`). Otherwise `#if canImport(AppKit)` / `#if os(macOS)` … `#else` (iOS) … `#endif`. Never `import AppKit` unguarded in a target that must compile for iOS.
- **iOS deployment target:** `.iOS(.v17)`.
- **No test was weakened.** The 190 tests are unchanged; they must pass on *both* macOS and the iOS simulator. If a test encodes a macOS-only assumption, that is a Phase 0 *finding* — report it, do not silently edit it.

- [ ] **Step 3: Gate the export renderer's AppKit usage**

`PortfolioExportRenderer.swift` uses `import AppKit` and `NSFont`/`NSColor`/`NSAttributedString` for PDF *text* drawing (the `CGContext` PDF core and the OOXML `.xlsx`/`.docx` ZIP writer are already cross-platform). Replace the AppKit imports/types with a cross-platform path:
- Change `import AppKit` to:
  ```swift
  #if canImport(AppKit)
  import AppKit
  typealias PlatformFont = NSFont
  typealias PlatformColor = NSColor
  #elseif canImport(UIKit)
  import UIKit
  typealias PlatformFont = UIFont
  typealias PlatformColor = UIColor
  #endif
  ```
- Replace `NSFont` → `PlatformFont`, `NSColor` → `PlatformColor` at every use (the `systemFont(ofSize:weight:)`, `monospacedDigitSystemFont(ofSize:weight:)`, `.labelColor`, and `NSColor(calibratedRed:…)` calls). For the calibrated-RGB initializer, use the cross-platform `init(red:green:blue:alpha:)` (UIColor has no `calibratedRed:`; NSColor accepts `init(red:…)` too) so one call site compiles on both. `NSAttributedString`/`NSMutableParagraphStyle` and the `.font`/`.foregroundColor` attribute keys are Foundation/cross-platform — verify the attribute key names compile on iOS (`.foregroundColor` is the key on both; on macOS AppKit it's also `.foregroundColor`).
- If any `NSAttributedString` drawing API (`draw(at:)`/`draw(in:)`) differs on iOS, wrap the *drawing* call (not the whole renderer) in `#if canImport(UIKit)` using `UIGraphicsPushContext(context)` around the iOS draw, since the AppKit `NSAttributedString.draw` needs an `NSGraphicsContext` current — on iOS push the `CGContext` via `UIGraphicsPushContext`. Keep the macOS path exactly as-is.
