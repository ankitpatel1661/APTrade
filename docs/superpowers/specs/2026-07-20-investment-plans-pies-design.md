# Investment Plans ("Pies") + DCA Backtest — Milestone 7 Design

**Date:** 2026-07-20
**Status:** Approved design, pre-implementation
**Platforms:** macOS, iPhone, Windows desktop, Android (all four, at parity)

## Context & goal

All four APTrade apps are at feature parity: watchlist, charts + indicators, paper-trading
portfolio with performance/risk analytics + export, news, price alerts, earnings calendar,
4-language settings. The portfolio answers "how are my holdings doing?" but not "how do I
invest consistently?"

Milestone 7 adds **Investment Plans** — target-weight baskets ("Pies", after Trading 212's
model) with recurring paper contributions, drift visibility, manual rebalancing, and a
historical DCA backtest. This is the first of a three-milestone arc decided 2026-07-20:

- **M7 — Investment Plans** (this spec)
- **M8 — Dividend & income engine** (unlocks DRIP inside Pies; dividend-adjusted returns)
- **M9 — Technical screener** over the bundled S&P 500 list (feeds Pies/watchlists)

## Decisions (settled with the user)

1. **Pies are slices of the one portfolio** — not sub-portfolios, not simulation-only.
   Pie buys draw from the same cash; holdings appear in the Holdings list; portfolio
   totals remain the single source of truth. (Trading 212 model.)
2. **Catch-up on launch** — schedules evaluate live while the app runs; on launch, missed
   contribution dates execute retroactively at each date's historical closing price.
   No OS background tasks.
3. **Self-balancing deposits + manual rebalance** — contributions skew toward underweight
   slices; a "Rebalance now" action (with confirm preview) is the only path that sells.
   No automatic selling, ever.
4. **DCA backtest lives inside the Pie create/edit wizard** — not a standalone tool.

## A. Domain model (pure; implemented twice: Swift `APTradeDomain` + Kotlin `shared/commonMain`)

### Entities

```
Pie
  id: String (UUID)
  name: String
  slices: [Slice]            // 1..N; targetWeights sum to exactly 100%
  schedule: ContributionSchedule?   // nil = manual-deposit-only Pie
  createdDay: String         // market-local yyyy-MM-dd (same shape as EarningsEvent.day)
  ledger: [SliceLedgerEntry] // shares acquired through this Pie, per symbol

Slice
  symbol: String
  assetKind: AssetKind
  targetWeight: Percentage   // construction-validated: all weights sum to 100%

ContributionSchedule
  amount: Money              // USD only in M7
  cadence: .weekly | .biweekly | .monthly
  nextDueDay: String         // yyyy-MM-dd; always a trading day (rolled forward)

SliceLedgerEntry
  symbol: String
  quantity: Decimal/BigDecimal  // fractional; both domains already use exact decimals
```

### Pure functions (the core; identical shape in both languages)

1. `distribute(contribution: Money, currentSliceValues: [Symbol: Money], targets: [Symbol: Percentage]) -> [Symbol: Money]`
   Self-balancing split: money flows preferentially into underweight slices until the Pie
   is back at target, then pro-rata by target weight.
   **Invariant (property-tested):** outputs sum *exactly* to `contribution`; no output is
   negative. Rounding remainder is assigned to the largest-target slice.

2. `rebalancePlan(currentSliceValues, targets) -> [RebalanceOrder]`
   `RebalanceOrder = (symbol, side: buy|sell, amount: Money)`. The sells/buys that restore
   target weights, net-zero cash. Preview-only until the user confirms.

3. `drift(currentSliceValues, targets) -> [Symbol: Percentage]`
   Signed actual-minus-target per slice, for the drift bars.

4. `dcaBacktest(slices, schedule, dailyCloses: [Symbol: [Day: Money]], calendar: MarketCalendar, range) -> BacktestReport`
   Replays the schedule over history: due dates generated by cadence, each rolled forward
   to the next trading day via the existing holiday-aware `MarketCalendar`; buys at that
   day's close using `distribute` against the simulated running slice values.
   `BacktestReport = (investedSeries, valueSeries, totalReturn, lumpSumComparison)` where
   `lumpSumComparison` invests the entire total on day one, same weights.

### Attribution & the manual-sell edge case

The Pie's `ledger` tracks shares acquired through the Pie. The portfolio remains the sole
authority on cash and aggregate positions. Pie contributions and rebalances execute through
the existing `BuyAsset`/`SellAsset` use cases, inheriting their shared-`Mutex` serialization
and order-fill notifications; each resulting transaction is tagged with the `pieId`
(transaction model gains an optional `pieId: String?` — absent for manual trades, so
existing persisted logs decode unchanged).

**Edge case (recorded):** if the user manually sells a symbol below the total quantity the
Pie ledgers claim, each affected Pie clamps its ledger to the actually-held quantity
(largest Pie clamps first) and the Pie detail shows a "manually adjusted" note. No
automatic buying-back.

A symbol MAY appear in multiple Pies and be held directly; ledgers keep attribution honest.

## B. Execution & scheduling

- A `ContributionDue(pieId, day)` event joins the `MarketActivityPlanner` event set
  (same pattern as `EarningsCheckDue`), evaluated by the existing per-platform
  market-activity coordinators. Executes at each due day's market open when the app is
  running.
- **Catch-up on launch:** on coordinator start, for each Pie compute all due days in
  `(lastExecuted, today]`; execute them in date order, each priced at that day's
  historical close (daily candles from the chart endpoint both stacks already call).
  Then advance `nextDueDay`.
- **Insufficient cash:** the entire contribution is skipped (no partial fills — keeps
  catch-up replay deterministic), recorded as `missed(reason: insufficientCash)` in the
  Pie's activity, and surfaced via a settings-gated notification.
- Notifications: `contribution executed` and `contribution skipped` join the existing
  notifier ports (`MarketEventNotifier` family), gated by a new settings toggle
  (default on), delivered per-platform exactly like earnings-day notifications.
- **Toggle semantics (adjudicated 2026-07-20, supersedes any narrower reading):** the
  `pieContributions` toggle gates **execution itself** on both paths (planner event
  emission and launch catch-up), not just notifications — toggle off acts as a global
  pause switch for all scheduled Pies. Mirrors the earnings-toggle pattern.

## C. Persistence

- New `PieStore` port (load/save all pies) in both application layers.
- macOS/iPhone: `UserDefaultsPieStore` (existing adapter pattern).
- Windows/Android: `FilePieStore` → `pies.json` in the OS config directory, implemented
  in `shared/jvmCommonMain` beside the other file stores; same JSON schema as the Swift
  side so the formats stay convergent.
- Transaction `pieId` tag persists through the existing portfolio stores; decoder treats
  it as optional for backward compatibility with existing saved portfolios.

## D. Presentation (all four platforms)

**No new tab.** A **"Plans"** section joins the Portfolio sub-switcher:
`Holdings · Allocation · Activity · Performance · Plans`.

- **Plans list** — cards: name, mini-donut of slices, current value, next contribution
  ("$200 · Mon Aug 3"), drift badge if any slice is >5pp off target. Empty state invites
  creating the first Pie.
- **Pie detail** — slice donut (reuses the existing donut component), per-slice rows
  (weight target vs. actual + drift bar), next-contribution line, "Contribute now" (manual
  one-off deposit through the same `distribute` path), "Rebalance now" (preview sheet:
  the `rebalancePlan` orders + Confirm/Cancel, honoring the Confirm Trades setting),
  per-Pie activity (contributions, rebalances, misses), edit, delete.
- **Create/edit wizard** — (1) name; (2) slices: search-to-add (existing search), weight
  editor with an equal-split helper, live 100% validation; (3) schedule: amount +
  weekly/biweekly/monthly + start day, or "no schedule"; (4) **Backtest** pane: invested
  vs. value curve over 1Y/3Y/5Y for the chosen schedule, total return, and the lump-sum
  comparison line. Backtest is display-only and never blocks saving (missing history →
  "insufficient history" note).
- Platform fit: macOS/iPhone share the SwiftUI implementation (`#if os(iOS)` fitting as
  usual — sheets on iPhone, inline panes on macOS). Desktop/Android implement the same
  anatomy in Compose, matching their existing Portfolio section idioms.
- **L10n:** ~35 new keys (section name, wizard, cadences, drift, backtest, notifications,
  misses) in EN/DE/IT/ES in both catalogs (Swift `L10n` + shared Kotlin `L10n`).
  Non-English values provisional pending native review, per the established convention.

## E. Dual-domain strategy & testing

The domain math is implemented twice (Swift + Kotlin), matching how portfolio/indicator/
risk math is already duplicated. Function names, shapes, and **test fixtures are kept
identical** on both sides so parity is mechanically checkable. (Converging macOS onto the
shared Kotlin core remains a recorded, separate decision — not part of M7.)

Tests, per side:
- **Property tests** on `distribute`: exact-sum invariant, non-negativity, underweight-
  first behavior, remainder assignment.
- **Fixture tests** on `dcaBacktest` (known candle fixtures → known report; holiday
  rolling across a Thanksgiving/Christmas window) and `rebalancePlan` (net-zero cash).
- **Use-case tests** with fake stores/repos: catch-up executes missed days in order at
  the right prices; insufficient-cash skip; ledger clamp on manual oversell; `pieId`
  round-trip through the transaction log.
- **ViewModel tests** per platform: wizard validation, drift display, preview flow.
- Suites to extend: macOS `swift test`, iOS simulator suite, `shared` commonTest,
  desktop test, android unit tests.

## Out of scope (M7)

- Dividend reinvestment (DRIP) — arrives with M8's dividend engine.
- Automatic rebalancing; any automatic selling.
- Daily cadence; multi-currency Pies; a standalone simulator screen.
- One-click "invest screener results into a Pie" — M9.

Same increment discipline as 6a–6e: shared/Kotlin core + desktop first or Swift side
first is a planning decision (writing-plans), but every platform lands within M7 before
the milestone closes, and the README gains a Plans feature block at merge.
